# OSINT Graph Intelligence Platform

> Визуальная система управления OSINT-данными: храните, связывайте и исследуйте сущности в виде графа.
>
> A visual OSINT data management system: store, link and explore entities as a graph.

---

## 🧭 Что это такое / What is this

**OSINT Graph Platform** — self-hosted веб-приложение для аналитиков, исследователей и специалистов по безопасности. Позволяет создавать базы данных объектов (людей, email, доменов, IP-адресов и других сущностей), строить между ними связи и визуализировать граф зависимостей.

Подходит для:
- Проведения OSINT-расследований
- Составления досье на физических и юридических лиц
- Анализа утечек данных
- Построения карт инфраструктуры
- Хранения и структурирования разведывательной информации

## 🚀 Быстрый старт / Quick Start

### Требования
- Python 3.9+
- Node.js 18+

### Запуск одной командой

```bash
cd osintgraphtool
chmod +x run.sh
./run.sh
```

Открывает:
- 🌐 Приложение: http://localhost:5173
- ⚙️ API: http://localhost:8000
- 📖 Документация API: http://localhost:8000/docs

### Продакшн-режим

```bash
./run.sh --prod
# Собирает фронтенд и отдаёт через бэкенд на одном порту 8000
```

### Параметры run.sh

| Флаг | Описание |
|------|----------|
| `--build` | Собрать фронтенд перед запуском |
| `--prod` | Продакшн: сборка + сервис на одном порту |
| `--port=PORT` | Порт бэкенда (по умолчанию 8000) |
| `FRONTEND_PORT=X` | Порт фронтенда (по умолчанию 5173) |

### Ручной запуск

```bash
# Backend
cd backend
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000

# Frontend (в другом терминале)
cd frontend
npm install
npm run dev
```

## 🔐 Шифрование данных / Data Encryption (v3.1)

Начиная с версии 3.1, все чувствительные поля сущностей (`value`, `metadata`, `notes`) шифруются перед записью в базу данных с помощью **AES-256-GCM**.

### Как это работает

1. При **регистрации** пользователя генерируется случайный 32-байтовый `enc_salt`.
2. Ключ AES-256 деривируется из `password_hash + enc_salt` через **PBKDF2-HMAC-SHA256** (200 000 итераций).
3. Перед записью в SQLite каждое поле шифруется; при чтении — расшифровывается прозрачно.
4. При **смене пароля** всё хранилище пользователя автоматически перешифровывается новым ключом.
5. Старые аккаунты без `enc_salt` продолжают работать (данные в открытом виде); шифрование активируется при первой смене пароля.

### Что это защищает

✅ **Дамп файла базы данных** — злоумышленник, получивший `.db` файл без знания `password_hash` и соли, не сможет прочитать данные сущностей.

⚠️ Это **серверное шифрование "в покое"** (at-rest), а не end-to-end. Сервер деривирует ключ при каждом запросе и может расшифровать данные активных пользователей. Если скомпрометирован сам сервер — данные читаемы.

## 📦 Резервное копирование / Backup

### Экспорт

`GET /api/backup/export` — Скачивает `.zip` архив:
- `data.json` — все сущности, связи и кастомные схемы (всегда в открытом тексте — расшифровываются при экспорте)
- `photos/` — встроенные фото, извлечённые из метаданных

### Импорт

`POST /api/backup/import` — Загружает `.zip` бэкап.

**Исправлено в v3.1:** Импортированные записи теперь корректно привязываются к пользователю (`user_id`) и шифруются его ключом. До исправления все импортированные записи имели `user_id = NULL` и были невидимы после импорта.

Правила:
- Записи с уже существующим ID пропускаются.
- Встроенные схемы (`is_builtin`) не импортируются.
- Связи пропускаются если одна из связанных сущностей отсутствует.

## 🐛 Исправленные баги и новые функции (v3.3.0)

### ✨ Новые функции

**Редактор типов связей**
- Добавлена отдельная страница «Типы связей» с полным CRUD-интерфейсом
- Встроенные типы автоматически создаются при первом входе
- Каждый тип связи поддерживает: emoji, цвет, описание, произвольные поля (text, date, url, number, entity)
- Редактирование и удаление как кастомных, так и встроенных типов

**Тип поля `entity`**
- При создании типа сущности можно добавить поле с типом `entity`
- При заполнении формы создания/редактирования — открывается поиск по всем сущностям
- Позволяет связывать сущности прямо внутри полей (например: сотрудники организации)

**Умный поиск при добавлении связи**
- Окно добавления связи теперь содержит поиск по имени сущности (вместо выпадающего списка из тысячи позиций)
- Также добавлен фильтр по типу связи — достаточно набрать несколько букв
- Список типов связей загружается из базы (включая кастомные)

### 🐛 Исправления

**Поля встроенных типов сущностей**
- Исправлена критическая ошибка: при редактировании встроенного типа (например, «Телефон») редактор теперь предзаполняется стандартными полями из пресетов
- После сохранения формы новые поля корректно отображаются при создании/редактировании сущности
- Больше нет расхождения между полями в схеме и полями в форме создания

**Дата рождения и формат даты**
- Устранён визуальный баг: рядом с полем ввода даты теперь отображается превью в выбранном пользователем формате (ДД.ММ.ГГГГ / ММ.ДД.ГГГГ / ISO)

**Фокус в настройках**
- Устранён сброс фокуса при вводе текста в разделе WebDAV настроек: секция вынесена в мемоизированный компонент (`React.memo`), что предотвращает перерисовку при изменении родительского состояния

**Шифрование файлов и фотографий**
- Вложения (файлы) теперь полностью шифруются с помощью AES-256-GCM перед записью в БД
- При скачивании данные расшифровываются автоматически
- Фотографии в metadata сущностей шифруются как часть JSON-поля (как и раньше)
- Все данные в покое защищены от утечки через raw-доступ к базе данных

## 🐛 Исправленные баги (v3.1.0)

| # | Баг | Исправление |
|---|-----|-------------|
| 1 | **ZIP импорт: данные невидимы после импорта** | `user_id` не выставлялся у Entity/Relationship/EntityTypeSchema. Исправлено в `backup.py`. |
| 2 | **Граф: пустая страница при переходе между персонами** | `rootId` хранился в локальном state и инициализировался из URL только один раз. При переходе по URL компонент не перемонтировался и state не обновлялся. **Исправлено:** `rootId` теперь читается напрямую из `useSearchParams()` на каждый рендер; смена персоны обновляет URL через `setSearchParams`. Контейнер Cytoscape теперь всегда в DOM (loading overlay вместо unmount). |
| 3 | **Экспорт: в архив попадали связи других пользователей** | Запрос в export теперь фильтрует Relationship по `user_id`. |
| 4 | **Отсутствие шифрования данных в базе** | Реализовано AES-256-GCM шифрование полей + деривация ключа из пароля через PBKDF2. |
